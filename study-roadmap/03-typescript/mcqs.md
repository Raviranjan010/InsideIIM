# TypeScript Practice MCQs

This document contains 20 multiple-choice questions covering TypeScript fundamentals, object structures, generics, and advanced utility types. These questions are designed to reinforce concepts key to developing robust applications and AI agent workflows, such as in LangGraph.js.

## Topic 1: Why TypeScript & Basic Types

### Question 1 (Scenario-Based)
Consider the following TypeScript code:
```typescript
let conversations: string = ["Hello", "World"];
```
Which of the following describes the compilation error generated by this code?
* A) Type 'string[]' is not assignable to type 'string'.
* B) Type 'string' is not assignable to type 'string[]'.
* C) Variable 'conversations' implicitly has an 'any' type.
* D) Type 'string' is not an array type.

**Correct Answer: A**

**Explanation:** Option A is correct because the variable is annotated as a single `string` but initialized with an array of strings, resulting in an assignability mismatch. Option B is incorrect because it incorrectly implies we are assigning a string to an array type; Options C and D are incorrect because the variable has an explicit (not implicit) type annotation, and TypeScript does not complain that string is not an array type but rather that the assigned value does not match the declaration.

### Question 2 (Conceptual)
Why is the `any` type generally discouraged when defining application and agent states in TypeScript?
* A) It compiles to a different JavaScript primitive at runtime, causing execution speed to drop significantly.
* B) It bypasses TypeScript’s compiler checks for that variable, re-introducing silent bugs and disabling editor autocomplete.
* C) It forces the compiler to throw a syntax error when the variable is passed to functions expecting primitive types.
* D) It automatically marks the variable as read-only, which prevents any subsequent updates to the agent state.

**Correct Answer: B**

**Explanation:** Option B is correct because `any` instructs TypeScript to disable type checking on the variable, meaning developers lose the safety nets of typo protection and auto-completion. Option A is incorrect because all TypeScript types (including `any`) are completely erased during compilation and do not affect JavaScript execution speed; Options C and D are incorrect because `any` is fully assignable to other types without compiler errors and does not make variables read-only.

### Question 3 (Conceptual)
Which of the following statements best describes how TypeScript code is executed by Node.js or web browsers?
* A) Modern browsers contain a built-in TypeScript engine that compiles and runs `.ts` files on the fly.
* B) Node.js executes TypeScript directly by using an internal transpilation layer that bypasses JavaScript engine limits.
* C) TypeScript must compile down to standard JavaScript first, which is then executed by the browser or Node.js.
* D) TypeScript is run within a special, secure CPU sandbox called the TSVM that isolates memory.

**Correct Answer: C**

**Explanation:** Option C is correct because TypeScript is a compile-time tool that compiles into clean JavaScript before runtime. Options A and B are incorrect because neither web browsers nor Node.js can execute TypeScript directly without transpilation; Option D is incorrect because TSVM is a fictional construct and TypeScript has no hardware-level VM.

### Question 4 (Scenario-Based)
What compilation error will TypeScript produce for the following assignment?
```typescript
let isAgentFinished: boolean = false;
isAgentFinished = "yes";
```
* A) Type 'boolean' is not assignable to type 'string'.
* B) Type 'string' is not assignable to type 'boolean'.
* C) Property 'isAgentFinished' is missing in type 'string'.
* D) Cannot assign to 'isAgentFinished' because it is a constant.

**Correct Answer: B**

**Explanation:** Option B is correct because the variable was declared as a `boolean` and cannot be assigned a string value like `"yes"`. Option A is incorrect because it describes the reverse assignment direction; Options C and D are incorrect because `isAgentFinished` is a variable rather than an object property, and it is declared using mutable `let` rather than `const`.

### Question 5 (Conceptual)
What is the correct and standard type annotation for a variable named `scores` initialized to an array of numbers?
* A) `let scores: Array(number) = [1, 2, 3];`
* B) `let scores: [number] = [1, 2, 3];`
* C) `let scores: number[] = [1, 2, 3];`
* D) `let scores: number = [1, 2, 3];`

**Correct Answer: C**

**Explanation:** Option C is correct because `number[]` is the standard syntax for denoting an array containing an arbitrary number of numbers. Option A is incorrect because `Array(number)` is syntactically invalid (the generic version is `Array<number>`); Options B and D are incorrect because `[number]` is a tuple requiring exactly one number, and `number` represents a single value rather than an array.

## Topic 2: Interfaces & Type Aliases

### Question 6 (Scenario-Based)
Observe the following code snippet:
```typescript
interface ToolInput {
  query: string;
  limit: number;
}
const input: ToolInput = {
  query: "Search for LangGraph tutorials"
};
```
What compilation error does this cause, and how can it be resolved?
* A) Property 'limit' is missing; resolve by adding `limit?: number` in the interface or supplying it in the object.
* B) Property 'query' is missing; resolve by adding a question mark after 'query' in the interface.
* C) Object literal specifies unknown properties; resolve by removing 'query' from the object.
* D) The variable 'input' must be defined using the 'type' keyword; resolve by changing interface to type.

**Correct Answer: A**

**Explanation:** Option A is correct because the interface requires the `limit` property, which is omitted in the object initialization, and can be fixed by either supplying it or marking it optional with `?`. Option B is incorrect because `query` is present in the object, not missing; Options C and D are incorrect because `query` is a known property on `ToolInput` and both interfaces and type aliases are valid for object shape definitions.

### Question 7 (Conceptual)
What is a key difference between `interface` and `type` regarding how they are extended or merged?
* A) Interfaces support declaration merging and the `extends` keyword, while type aliases use intersection (`&`) and cannot be redeclared.
* B) Type aliases support declaration merging and the `extends` keyword, while interfaces use intersection (`&`) and cannot be redeclared.
* C) Interfaces can only define primitive types and cannot be extended, while type aliases can define object shapes.
* D) Interfaces are converted to classes in JavaScript, whereas type aliases are completely removed during compilation.

**Correct Answer: A**

**Explanation:** Option A is correct because interfaces support extending via `extends` and can be declared multiple times to merge properties, whereas type aliases cannot be reopened and must combine via intersection. Option B is incorrect because it describes the reverse behavior of types and interfaces; Options C and D are incorrect because interfaces define object shapes (not primitives) and both types and interfaces are completely erased from compiled JavaScript.

### Question 8 (Scenario-Based)
Consider this code:
```typescript
type AgentRole = "researcher" | "writer" | "editor";
interface Agent {
  name: string;
  role: AgentRole;
}
const firstAgent: Agent = {
  name: "DocBot",
  role: "reviewer"
};
```
What error does the TypeScript compiler report?
* A) Type '"reviewer"' is not assignable to type 'AgentRole'.
* B) Property 'role' is missing in type 'Agent'.
* C) Duplicate identifier 'AgentRole'.
* D) Cannot find name 'reviewer'.

**Correct Answer: A**

**Explanation:** Option A is correct because the value `"reviewer"` is not one of the allowed string literals defined in the `AgentRole` union. Option B is incorrect because `role` is provided in the object initializer; Options C and D are incorrect because `AgentRole` is defined only once, and `"reviewer"` is a string literal rather than an undefined variable name.

### Question 9 (Conceptual)
How do you declare a property in an interface as optional, indicating that it does not have to be provided?
* A) Prefix the property name with an asterisk (`*`).
* B) Use the `optional` keyword before the property name.
* C) Place a question mark (`?`) after the property name.
* D) Assign a default value inside the interface definition.

**Correct Answer: C**

**Explanation:** Option C is correct because placing a `?` after the property name (e.g. `isFinished?: boolean`) makes it optional in TypeScript. Options A and B are incorrect because `*` and `optional` are not recognized syntaxes for optional properties; Option D is incorrect because interfaces only declare types and cannot contain default values.

### Question 10 (Conceptual)
Which of the following is the correct way to declare a union type `ModelSize` that can be either "small", "medium", or "large"?
* A) `interface ModelSize { small: string; medium: string; large: string; }`
* B) `type ModelSize = "small" | "medium" | "large";`
* C) `type ModelSize = "small" & "medium" & "large";`
* D) `let ModelSize: "small" | "medium" | "large";`

**Correct Answer: B**

**Explanation:** Option B is correct because the `type` keyword combined with the union operator `|` defines a type alias that can hold any of the specified string literals. Option A is incorrect because it defines an object interface requiring all three fields; Option C is incorrect because an intersection `&` of literals is impossible and results in `never`; Option D is incorrect because it declares a runtime variable instead of a type alias.

## Topic 3: Generics

### Question 11 (Conceptual)
What is the primary purpose of using Generics in TypeScript?
* A) To disable static typing and allow variables to dynamically change types at runtime.
* B) To create reusable, flexible components and interfaces that work with multiple types while maintaining type safety.
* C) To force the compiler to convert all variables to their generic JavaScript equivalent.
* D) To define global configuration variables that can be accessed across different modules.

**Correct Answer: B**

**Explanation:** Option B is correct because generics allow developers to pass types as parameters, making components highly reusable without sacrificing type check safety. Option A is incorrect because generics enforce strict types once instantiated, unlike dynamic typing; Options C and D are incorrect because generics do not affect runtime execution and are not used for global variables.

### Question 12 (Scenario-Based)
Analyze the following code:
```typescript
interface ResponseContainer<Data> {
  status: number;
  data: Data;
}
const myResponse: ResponseContainer<string> = {
  status: 200,
  data: 100
};
```
What compilation error will occur here?
* A) Type 'number' is not assignable to type 'string'.
* B) Property 'data' is missing in type 'ResponseContainer'.
* C) Generic type 'ResponseContainer' requires two type arguments.
* D) Cannot assign type 'string' to generic type parameter 'Data'.

**Correct Answer: A**

**Explanation:** Option A is correct because the generic parameter `Data` was instantiated as `string`, which requires `data` to be a string, but a number (`100`) was assigned. Option B is incorrect because `data` is present in the object; Options C and D are incorrect because the interface only takes one type argument and `string` is a valid type parameter.

### Question 13 (Conceptual)
In LangGraph.js, when you see `new StateGraph<MyAgentState>(...)`, what role does `<MyAgentState>` play?
* A) It is a subclass constructor that instantiates `MyAgentState` inside the graph.
* B) It is a generic parameter that tells the graph exactly what state shape to enforce for nodes and channels.
* C) It is a compilation flag that enables dynamic type assertion at runtime.
* D) It is an argument containing the default values for the graph state.

**Correct Answer: B**

**Explanation:** Option B is correct because `<MyAgentState>` is a generic type argument that instructs `StateGraph` to type-check all state transitions against the `MyAgentState` shape. Option A is incorrect because angle brackets represent type arguments rather than class constructors; Options C and D are incorrect because TypeScript types are erased at runtime and cannot provide default values or runtime assertions.

### Question 14 (Scenario-Based)

```typescript
interface NodeExecutionResult<State> {

interface TextState {

const execution: NodeExecutionResult<TextState> = {
```

What error is thrown by the compiler?

* B) Property 'updatedState' is missing in type 'NodeExecutionResult'.
* C) Type 'string' is not assignable to type 'TextState'.
* D) Object literal may only specify known properties, and 'success' is not valid.

**Correct Answer: A**

**Explanation:** Option A is correct because `updatedState` is typed as `TextState` (due to the generic argument), but the assigned object is missing the required `text` property and includes an unknown key `invalidKey`. Option B is incorrect because `updatedState` is defined; Options C and D are incorrect because the value is an object (not a string) and `success` is a valid property on the `NodeExecutionResult` interface.

## Topic 4: Utility Types & Advanced Patterns

### Question 16 (Conceptual)
Which of the following best describes the effect of the `Partial<T>` utility type?
* A) It removes all properties from `T` except for those explicitly specified.
* B) It makes all properties of `T` optional by appending `?` to each property name.
* C) It makes all properties of `T` read-only.
* D) It combines all properties of `T` with another type.

**Correct Answer: B**

**Explanation:** Option B is correct because `Partial<T>` constructs a type where all properties of `T` are set to optional, which is useful for state updates. Option A is incorrect because that describes `Pick<T, Keys>`; Options C and D are incorrect because making properties read-only is done by `Readonly<T>`, and combining types uses intersection (`&`).

### Question 17 (Scenario-Based)
Analyze the following code:
```typescript
interface User {
  id: string;
  name: string;
  email: string;
}
type UserUpdate = Omit<User, "id">;
const update: UserUpdate = {
  id: "123",
  name: "Alice",
  email: "alice@example.com"
};
```
What compilation error is triggered?
* A) Property 'id' does not exist on type 'UserUpdate'.
* B) Property 'name' is missing in type 'UserUpdate'.
* C) Type 'string' is not assignable to type 'Omit<User, "id">'.
* D) The variable 'update' must be initialized with all properties of 'User'.

**Correct Answer: A**

**Explanation:** Option A is correct because the `Omit<User, "id">` type excludes the `id` property, making it an invalid key to assign in the object literal. Option B is incorrect because `name` is present in the object; Options C and D are incorrect because `update` is an object shape (not a string) and `Omit` removes the need to supply the excluded properties.

### Question 18 (Conceptual)
Which utility type is designed to construct an object type whose property keys are of one type and whose values are of another?
* A) `Pick<Keys, Type>`
* B) `Omit<Keys, Type>`
* C) `Record<Keys, Type>`
* D) `Partial<Keys, Type>`

**Correct Answer: C**

**Explanation:** Option C is correct because the `Record` utility type creates a dictionary structure mapping key types to value types (e.g. `Record<string, number>`). Options A and B are incorrect because `Pick` and `Omit` extract or exclude properties from an existing object type; Option D is incorrect because `Partial` only takes a single type argument to make properties optional.

### Question 19 (Scenario-Based)
Consider the following narrowing attempt:
```typescript
function processInput(input: string | string[]) {
  if (typeof input === "object") {
    console.log(input.join(", "));
  } else {
    console.log(input.toUpperCase());
  }
}
```
Why is this code unsafe and what potential runtime issue exists?
* A) If 'input' is an array, `typeof input === "object"` will evaluate to false, causing it to fall to the `else` block and crash.
* B) If 'input' is `null`, `typeof` will evaluate to `"object"` and attempt to call `join` on it, crashing at runtime.
* C) The compiler will reject the code because `toUpperCase` is not available on arrays.
* D) The union type `string | string[]` is invalid syntax and cannot compile.

**Correct Answer: B**

**Explanation:** Option B is correct because `typeof` returns `"object"` for arrays as well as `null` and plain objects, making it an unsafe way to verify if a variable is an array without using `Array.isArray()`. Option A is incorrect because `typeof` on an array does return `"object"`, so it would run the `if` block, not the `else` block; Options C and D are incorrect because TypeScript correctly infers `input` is a `string` in the `else` block, and union types are perfectly valid.

### Question 20 (Scenario-Based)
Look at this code:
```typescript
interface SearchQuery { query: string; }
interface AuthorDetails { author: string; }
type Action = SearchQuery | AuthorDetails;

function handleAction(action: Action) {
  console.log(action.author);
}
```
What compilation error does this generate?
* A) Property 'author' does not exist on type 'Action'. Property 'author' does not exist on type 'SearchQuery'.
* B) Type 'Action' is not assignable to type 'AuthorDetails'.
* C) Union types cannot be used as function arguments.
* D) Property 'query' is missing in type 'AuthorDetails'.

**Correct Answer: A**

**Explanation:** Option A is correct because `Action` is a union, and since the compiler cannot guarantee the runtime value is not a `SearchQuery` (which lacks `author`), it prevents accessing `author` directly. Option B is incorrect because `action` is not being assigned to a variable of type `AuthorDetails`; Options C and D are incorrect because union types can be parameters and the compiler is concerned about the access safety of `author`, not the missing `query` in `AuthorDetails`.
